# 体考体测同引擎双业务设计

## 1. 设计结论

- 体考与体测共用一套评分引擎：本质都是“成绩 -> 分值/等级”规则匹配。
- 体考与体测分为两个业务板块：录入、分析、报表、标准管理页面完全分离。
- 规则导入框架复用，但标准版本、业务口径、菜单与权限分别维护。

---

## 2. 为什么必须双业务

1. 规则口径不同
- 体考：考试导向，围绕及格/优秀/满分和总分。
- 体测：体质结构导向，围绕单项体质指标与长期变化。

2. 管理流程不同
- 体考：按考试批次组织。
- 体测：按体测批次/学期组织。

3. 展示目标不同
- 体考强调考试结果与阈值达成。
- 体测强调体质结构变化与学校健康监测。

## 4. 统一评分引擎模型

## 4.1 核心字段

- `biz_type`：`pe_exam` / `fitness_test`
- `standard_version_id`
- `item_id`
- `gender`
- `grade_stage`
- `direction`：`higher_better` / `lower_better`
- `raw_min`、`raw_max`
- `score_value`
- `level`
- `priority`

## 4.2 规则匹配流程

1. 按 `biz_type + standard_version_id + item_id + gender + grade_stage` 过滤规则集。
2. 按 `direction` 匹配成绩区间。
3. 返回 `score_value + level`。
4. 冲突规则按固定策略“就低不就高”。

## 6. API 分组建议

- 体考：`/vadmin/pe/*`
- 体测：`/vadmin/fitness/*`
- 通用评分引擎：仅 service 内部复用，不直接暴露。

示例：

- `POST /vadmin/fitness/scores/import`
- `GET /vadmin/fitness/analysis/overview`
- `GET /vadmin/fitness/analysis/class`
- `GET /vadmin/fitness/analysis/grade`
